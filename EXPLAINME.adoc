The README makes claims. This file backs them up with concrete data structures and critical algorithms.

== Claim 1: SQLite-Backed Knot Database with Automatic Invariant Computation

From README:
____
Storage: SQLite-backed, WAL mode for concurrent reads. Knots stored as serialised Gauss codes with invariants computed on insert. Schema migrations handled automatically.
____

**How it's proven:**
- **Core struct**: `GaussCode` type defined in `src/types.jl` (2.5KB)
- **Record type**: `KnotRecord` struct stores: name, Gauss code, crossing_number, writhe, gauss_hash (SHA-256 content-addressing), jones_polynomial, genus, seifert_circle_count
- **Database abstraction**: `SkeinDB` exported at line 44; created via `SkeinDB("knots.db")` (README line 32)
- **Storage operation**: `store!(db, name, gauss_code, metadata=...)` (exported line 47) performs: (1) validates Gauss code, (2) computes invariants in `src/invariants.jl`, (3) inserts into SQLite with schema v3 (auto-migrates if needed), (4) logs to WAL for concurrent reads
- **Critical path**: `src/storage.jl` (11KB) implements SQLite lifecycle with `_get_schema_version` + `_migrate_vN_to_vM` pattern (handles v1→v2→v3 migrations transparently)
- **Invariant computation**: `crossing_number`, `writhe`, `gauss_hash` computed standalone in `src/invariants.jl` (9KB); Jones polynomial computed via KnotTheory.jl extension when loaded

**Caveat:** Jones polynomial computation requires KnotTheory.jl (weakdep, optional). Without it, only standalone invariants (crossing, writhe, hash) are computed. Alexander polynomial NOT implemented—would require crossing chirality data not stored in basic Gauss codes.

== Claim 2: Equivalence Checking via Cyclic Rotation, Mirroring, and Reidemeister I Simplification

From README:
____
Equivalence checking: Detect equivalent knot diagrams via cyclic rotation, relabelling, mirroring, and Reidemeister I simplification. Find equivalent or isotopic knots in the database with find_equivalents and find_isotopic.
____

**How it's proven:**
- **Core functions**: `is_equivalent(g1, g2)` (diagram-level), `is_isotopic(g1, g2)` (topology-level) exported lines 57
- **Database search**: `find_equivalents(db, g)` and `find_isotopic(db, g)` exported line 59
- **Rotation invariance**: `canonical_gauss(g)` normalizes to smallest cyclic rotation + relabelling (exported line 58)
- **Mirror support**: `mirror(g)` computes mirror image; `is_amphichiral(g)` checks if knot equals its mirror (lines 57-58)
- **Reidemeister simplification**: `simplify_r1(g)` removes crossing pairs; `simplify_r2(g)` simplifies 2-crossing passes; `simplify(g)` applies R1+R2 iteratively (line 57)
- **Topology equivalence**: `is_isotopic` internally calls `simplify_r1` then checks equivalence (rotation + relabelling only)
- **Critical path**: `src/invariants.jl` (9KB) implements canonical form computation; `src/polynomials.jl` (8KB) computes Jones polynomial for invariant-based deduplication

**Caveat:** Reidemeister moves beyond R1 and R2 are not implemented. R3 moves (triangle rearrangement) would require more sophisticated crossing tracking. The `is_isotopic` guarantee is therefore sound only for knots distinguishable by R1+R2 simplification plus Jones polynomial.

== Technology Choices

[cols="1,2"]
|===
| Technology | Purpose

| **Julia** | Core knot data structures, invariant computation, equivalence checking
| **SQLite.jl** | Persistent storage with WAL mode; direct cursor iteration (never `collect()` due to SQLite 1.8 finalization issues)
| **KnotTheory.jl (weakdep)** | Optional: Jones polynomial computation, PlanarDiagram conversion, Reidemeister move extension
| **SHA.jl** | Content-addressed hashing of Gauss codes for deduplication
|===

== Dogfooded Across The Account

Skein.jl is the **knowledge base** for knot theory research in the ecosystem:
- **KnotTheory.jl**: Provides computation engine; Skein provides persistence layer
- **Cliodynamics.jl**: May use Skein for historical topology mapping (invariant queries)
- **PostDisciplinary.jl**: Knot records are `LinkedEntity` objects with `kind=:knot`

Same database abstraction pattern (schema versioning, cursor iteration) used in `VeriSimDB`, `Lithoglyph`, and `rrecord-verity`.

== Critical Invariants (What Must Not Break)

1. **Gauss hash uniqueness**: `gauss_hash(g) == gauss_hash(g')` iff the Gauss codes represent the same diagram (rotation/relabelling applied)
2. **Equivalence is symmetric**: `is_equivalent(g1, g2) == is_equivalent(g2, g1)`
3. **Isotopy is transitive**: If `is_isotopic(g1, g2)` and `is_isotopic(g2, g3)`, then `is_isotopic(g1, g3)`
4. **Canonical form is idempotent**: `canonical_gauss(canonical_gauss(g)) == canonical_gauss(g)`
5. **Mirror involution**: `mirror(mirror(g)) == g`
6. **Cursor iteration safety**: Never call `collect()` on SQLite result sets; iterate directly to avoid finalization issues
7. **Schema persistence**: Schema version must not decrease; migrations are forward-only
