= Skein.jl — Backend Abstraction Layer
:toc:

== Purpose

Coprocessor-dispatch glue that decouples Skein's compute-intensive paths from
execution hardware. The same `AcceleratorGate` mechanism used in `KnotTheory.jl`
is applied here, providing a uniform pattern across the KRL stack.

== Files

[cols="1,3"]
|===
| File | Purpose

| `abstract.jl`
| Five dispatch hooks with Julia fallbacks + `generate_self_healing_hooks`
  wiring for coprocessor override with automatic fall-through.
|===

== Dispatch hooks

[cols="1,3"]
|===
| Hook | Used for

| `backend_invariant_compute`
| Batch invariant computation for `backfill_gauss_canonical!` and bulk imports.
  The Julia fallback computes each invariant sequentially; a GPU backend could
  parallelise across thousands of knots.

| `backend_polynomial_eval`
| Jones/Kauffman polynomial evaluation at a point. Shared with KnotTheory
  hook (same name, different module scope).

| `backend_equivalence_check`
| `is_equivalent` / `is_isotopic` pairwise checks. Naïve implementation is O(n²);
  a coprocessor backend can use canonical invariant fingerprinting.

| `backend_batch_query`
| Large result-set queries where the SQLite layer is the bottleneck.
  A backend could materialise a hot cache or use an in-memory index.

| `backend_simplify`
| Gauss-code Reidemeister simplification (R1/R2 reduction).
  Delegated here so a coprocessor can vectorise across a bulk queue.
|===

== Adding a new backend

. Implement methods dispatching on your concrete `AcceleratorGate` backend type:
+
[source,julia]
----
using AcceleratorGate: MyCustomBackend
Skein.backend_equivalence_check(::MyCustomBackend, a::GaussCode, b::GaussCode) = ...
----
. `generate_self_healing_hooks` in `abstract.jl` handles routing and fallback.
   No changes to `abstract.jl` needed.
